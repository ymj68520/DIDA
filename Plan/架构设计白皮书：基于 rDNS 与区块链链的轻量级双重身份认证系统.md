# 架构设计白皮书：基于 rDNS 与联盟链的轻量级双重身份认证系统

## 一、 架构愿景与设计哲学

本系统旨在解决传统DNSSEC/DANE在跨域认证中存在的报文臃肿（UDP 截断）、信任链过长以及时延放大等工程性缺陷。架构设计的核心哲学是**"链上链下解耦，I/O 与计算分离"**：
1. **数据面极简**：通过 rDNS TXT 记录实现带外（Out-of-Band）哈希指针寻址，将网络开销降至最低。TXT 记录仅承载 `TxID`（链上凭证索引）与 `Sig_Sub`（节点对自身 IP 和 TxID 的静态自签名），两字段语义清晰、相互独立。
2. **控制面异步**：利用 Rust 异步调度，引入 **In-Flight Task Tracker** 防御 TCP 重传冲击，将网络 I/O 等待与密码学计算解耦。
3. **信任根下沉**：联盟链链上完整保存 `Cert_IP`（IP 归属凭证主体）与顶级机构授权签名 `Sig_Top`，本地预置信任锚公钥 $PK_{Top}$，构成两层不可伪造的信任链。结合 **WebSocket 事件监听** 与本地多级缓存，实现秒级撤销实时性与常数级核验耗时。

## 二、 系统拓扑与四大核心模块设计

本系统物理/逻辑上划分为四个高度解耦的微服务模块，通过标准网络协议（TCP/UDP/HTTP）进行交互，确保单机虚拟化环境下的高保真度。

### 模块一：超高性能验证网关 (Authentication Gateway)
这是系统的核心数据面引擎，拦截底层流量并执行双重验签。
* **技术底座**：Rust (Edition 2021) 配合 `tokio` 异步运行时。
* **网络拦截层**：基于 `libnetfilter_queue`。在内核态挂起 TCP SYN 报文，只将必要的 Header 和 IP 载荷零拷贝（或极低开销拷贝）传递给用户态 Rust 进程，避免全量报文解析带来的延迟。
* **本地信任锚**：网关启动时从本地配置文件加载 $PK_{Top}$（顶级权威机构的公钥），该公钥不依赖任何网络查询，是执行 $V_1$ 验证的不动根基。
* **异步调度模型**：
  * **I/O 线程池**：专职负责 `nfq` 报文读取、`hickory-resolver` DNS 请求、以及 `alloy` RPC 请求。
  * **计算线程池**：隔离运行 `k256` 签名引擎，处理 $V_1$（用本地 $PK_{Top}$ 验证链上凭证的权威背书）和 $V_2$（用凭证中的 $PK_{Sub}$ 验证节点自签名）。
  * **XDP/eBPF 加速层（白名单初筛）**：在内核驱动层对**已缓存的合法 IP 集合**执行快速 MAP 查找，直接放行白名单流量，绕过用户态处理。注意：ECDSA 验签等高计算量密码学操作受限于内核 eBPF 验证器的指令数和内存约束，**无法在 XDP 层执行**。XDP 层的职责仅限于比对已建立信任的 IP 白名单，$V_1$ 和 $V_2$ 的密码学验证始终在用户态计算线程池中完成。
* **本地状态机 (Cache)**：内置 `moka` 缓存。采用 **refresh_after_write** 策略，并订阅链上事件实现主动失效。

### 模块二：联盟链状态机与信任锚点 (Consortium Blockchain)
负责存储所有域名的公钥凭证、顶级授权签名和吊销状态，提供具备全局一致性的去中心化 PKI。
* **节点基础设施**：采用 Foundry 体系的 Anvil 本地节点。配置为"即时出块"（Zero Block Time）模式，最大化模拟极高 TPS 的联盟链（如 Hyperledger Fabric 或以太坊 PoA 侧链）的响应特性。**注意**：Anvil `eth_call` 的读取时延与真实联盟链共识机制（PBFT、Raft 等）的提交时延存在差异，本实验测量的是只读查询路径，不涉及共识写延迟，论文 Discussion 节需对此差异加以说明。
* **数据结构设计 (Registry)**：
  * **核心映射**：`mapping(bytes32 => ChainRecord) public registry`，以 `TxID` 为键。
  * **完整链上结构（`ChainRecord`）**：
    * `certIP`：`Cert_IP` 凭证主体，包含 IP 前缀、`PK_Sub`（公钥）、颁发时间戳、吊销状态位，是 $V_1$ 验证的被签名对象。
    * `sigTop`：`bytes` 类型，存储顶级权威机构（如 IANA/RIR 对应的联盟链权威节点）对 `Hash(Cert_IP)` 的 ECDSA 签名 $Sig_{Top}$，是 $V_1$ 的核心验证素材。
  * **丢失 `Sig_Top` 的安全后果**：若链上不存储 `Sig_Top`，任何拥有合约写入权限的节点均可伪造 `TxID → 任意公钥` 映射，验证网关无法察觉，整个双重信任链退化为无信任根的普通 PKI。
* **时间复杂度说明**：通过扁平 `mapping` 结构，底层 EVM 通过 Merkle Patricia Trie（MPT）寻址。MPT 查找的严格复杂度为 $O(k)$，其中 $k$ 为键的长度。由于 `TxID` 为固定长度的 32 字节（64 个十六进制字符），$k$ 是常数，因此查询路径深度与链上状态总量无关，整体查询复杂度与数据规模解耦，表现为常数级别，这是支撑论文 Exp-3 的核心论点。

### 模块三：带外反向 DNS 基础设施 (rDNS Index Source)
作为数据面的引路人，提供极致压缩的认证报文，仅承载两个关键字段。
* **服务端引擎**：容器化的 BIND9 服务端，监听本地隔离环回地址 `127.0.0.2:53`（规避 `systemd-resolved` 默认占用 `127.0.0.53` 导致的端口冲突）。
* **索引结构规范**：
  * 在 `.in-addr.arpa` 逆向解析域中配置自定义 TXT 记录。
  * 记录格式：协议版本号 + 32 字节 `TxID`（哈希指针） + `Sig_Sub`（节点对 `IP + TxID` 的静态自签名，用于 $V_2$ 校验）。总载荷严格控制在 150 字节以内，远低于 512 字节 UDP 截断阈值。
  * **`Sig_Sub` 的语义边界**：该签名为节点预计算并存储于 Zone File 中的静态签名，所有查询者共享同一份值。其提供的保护是"证明 IP 地址与 TxID 的绑定关系由持有 $SK_{Sub}$ 的实体背书"，而非会话级别的防重放。真正的防重放保护依赖 DNS TTL 窗口（TTL 到期后 DNS 缓存失效）和凭证吊销机制，论文中需明确说明该设计的安全假设边界。

### 模块四：实验编排与网络保真模块 (Experimental Testbed)
为了向顶刊审稿人证明实验环境的严谨性，必须引入网络仿真和极限压测机制。
* **流量发生器（Exp-5）**：采用 Rust 编写的自定义 TCP 并发客户端，模拟**完整 TCP 三次握手**的并发认证连接（而非 SYN Flood）。只有完成三次握手，网关的 $V_2$ 验签才能走到终态、下发 `NF_ACCEPT`，Exp-5 测出的 TPS 才真实反映完整认证流程的吞吐能力。
* **网络损伤模拟 (Network Emulation)**：
  * 引入 Linux `tc` (Traffic Control) 结合 `netem` 模块，基于 `iptables fwmark` 对特定端口实施精细化延迟注入。
  * **针对 DNS 通道 (UDP 53)**：注入 $20 \sim 50$ ms 正态分布延迟。
  * **针对区块链 RPC 通道 (TCP 8545)**：注入 $50 \sim 150$ ms 延迟 + $0.1\%$ 随机丢包。
  * **规则优先级保障**：必须明确 `iptables NFQUEUE` 规则与 `tc fwmark` 规则的配置顺序，建议先配置 `tc` 延迟注入规则，再配置 `NFQUEUE` 拦截规则，并通过基准测试验证两套规则无叠加干扰。

## 三、 核心协议执行流水线 (Critical Path)

当一个客户端尝试向目标服务器发起连接时，系统的数据流转严格遵循以下生命周期：

> **核心原则**：$V_1$ 所需的 $PK_{Top}$ 来自网关本地预置，不需要网络查询；$V_2$ 所需的 $PK_{Sub}$ 来自链上 $Cert_{IP}$，必须先完成区块链查询。因此流水线中**不存在"先于区块链查询的 $V_1$"**，正确顺序如下：

1. **拦截与挂起**：内核 `iptables` 捕获目标出向的 TCP SYN 报文，挂入 Netfilter 队列，网关获取目标 IP。
2. **带外寻址**：网关向 BIND9 发起 `IP.in-addr.arpa` 的 TXT 查询，获取 `TxID` 与 `Sig_Sub`。
3. **状态检索与缓存介入**：
   * 网关以 `TxID` 访问本地 `moka` 缓存。
   * **Cache Miss**：向 Anvil 节点发起只读 RPC 拉取链上完整 `ChainRecord`（`Cert_IP` + `Sig_Top`），并回写缓存。
   * **Cache Hit**：直接进入下一步（消除 RPC 网络开销）。
4. **第一重防御 ($V_1$，权威背书校验)**：
   * 使用**本地预置的** $PK_{Top}$ 验证：$Verify(PK_{Top},\ Sig_{Top},\ Hash(Cert_{IP}))$。
   * 同时校验：$T_{current} \le Cert_{IP}.expiration$（凭证未过期）且目标 IP 属于 $Cert_{IP}.ip\_prefix$（IP 归属合法）。
   * **失败则立即下发 `NF_DROP`**，阻断连接（此步防御链上凭证被伪造）。
5. **第二重防御 ($V_2$，节点控制权校验)**：
   * 从 $Cert_{IP}$ 中提取 $PK_{Sub}$。
   * 验证 TXT 记录中的 `Sig_Sub`：$Verify(PK_{Sub},\ Sig_{Sub},\ IP \| TxID)$。
   * **失败则下发 `NF_DROP`**，阻断连接（此步防御持有链上凭证但无私钥控制权的伪冒）。
6. **裁决释放**：双重验签均通过，网关向内核下发 `NF_ACCEPT`，TCP 握手继续，连接建立。

## 四、 顶刊实验数据保障策略

为了确保你在 Exp-1 到 Exp-5 中拿到预期数据，架构设计已内置了以下保障：
* **对标 Exp-1 (时延拆解)**：系统在流水线的每个阶段（DNS、RPC、$V_1$、$V_2$）均埋入纳秒级探针 (`std::time::Instant`)。由于 Rust 没有垃圾回收期的 Stop-The-World 现象，采集到的拆解时延将能够完美拟合总端到端时延。
* **对标 Exp-2 (报文大小)**：BIND9 的精简 TXT 记录天然规避了 UDP 512 字节截断阈值，这在 Wireshark 抓包数据中将呈现极其直观的断层式对比。注意实测 DNSSEC 响应大小通常在 500~1200 字节范围，应以实测数据为准而非预先假设超过 1500 字节。
* **对标 Exp-5 (高并发吞吐)**：利用 `tokio` 的轻量级协程配合独立计算线程池，实验使用**完整 TCP 三次握手**的并发客户端测量真实认证 TPS；另设禁用 DNS 缓存组（测量最坏情况）和启用 DNS 缓存组（贴近真实部署），两组数据均在论文中呈现并说明各自的现实含义。

## 五、 实施路径与执行建议

建议按照以下三个阶段进行工程落地：
1. **基建与连通阶段**：部署 BIND9（监听 `127.0.0.2:53`，需在实验脚本中停止 `systemd-resolved` 或将其 stub resolver 配置为转发）与 Anvil 节点，写入静态测试数据（含 `Cert_IP` + `Sig_Top`）。编写简单的 Python/curl 脚本验证 DNS 解析和 RPC 调用的连通性。
2. **核心网关开发阶段**：集中精力用 Rust 开发 `Auth Gateway`。本地预置 $PK_{Top}$；先实现 `nfq` 拦截与无条件放行，再逐步接入异步 DNS 查询、缓存模块，最后按正确流水线顺序接入 $V_1$（使用 $PK_{Top}$ 验证 $Sig_{Top}$）和 $V_2$（使用 $PK_{Sub}$ 验证 $Sig_{Sub}$）密码学验签模块。
3. **仿真与压测阶段**：引入 `tc netem` 设置分端口网络延迟（先配置 `tc` 规则再配置 `NFQUEUE` 规则），使用完整握手的压测工具打满流量。开启 `perf` 或 `flamegraph` 火焰图定位性能热点，导出 CSV 实验数据进行绘图。
