# 架构设计白皮书：基于 rDNS 与联盟链的轻量级双重身份认证系统

## 一、 架构愿景与设计哲学

本系统旨在解决传统DNSSEC/DANE在跨域认证中存在的报文臃肿（UDP 截断）、信任链过长以及时延放大等工程性缺陷。架构设计的核心哲学是**“链上链下解耦，I/O 与计算分离”**：
1. **数据面极简**：通过 rDNS TXT 记录实现带外（Out-of-Band）哈希指针寻址，引入时间戳签名防御重发，将网络开销降至最低。
2. **控制面异步**：利用 Rust 异步调度，引入 **In-Flight Task Tracker** 防御 TCP 重传冲击，将网络 I/O 等待与密码学计算解耦。
3. **信任根下沉**：利用联盟链作为全局状态机，结合 **WebSocket 事件监听** 与本地多级缓存，实现秒级撤销实时性与常数级核验耗时。



---

## 二、 系统拓扑与四大核心模块设计

本系统物理/逻辑上划分为四个高度解耦的微服务模块，通过标准网络协议（TCP/UDP/HTTP）进行交互，确保单机虚拟化环境下的高保真度。

### 模块一：超高性能验证网关 (Authentication Gateway)
这是系统的核心数据面引擎，拦截底层流量并执行双重验签。
* **技术底座**：Rust (Edition 2021) 配合 `tokio` 异步运行时。
* **网络拦截层**：基于 `libnetfilter_queue`。在内核态挂起 TCP SYN 报文，只将必要的 Header 和 IP 载荷零拷贝（或极低开销拷贝）传递给用户态 Rust 进程，避免全量报文解析带来的延迟。
* **异步调度模型**：
  * **I/O 线程池**：专职负责 `nfq` 报文读取、`hickory-resolver` DNS 请求、以及 `alloy` RPC 请求。
  * **计算线程池**：隔离运行 `k256` 签名引擎，处理 $V_1$（带时间戳校验）和 $V_2$。
  * **核心优化**：引入 **XDP/eBPF 加速层** 初筛白名单，通过中间件模式支持 **Ablation Study** 消融实验。
* **本地状态机 (Cache)**：内置 `moka` 缓存。采用 **refresh_after_write** 策略，并订阅链上事件实现主动失效。

### 模块二：联盟链状态机与信任锚点 (Consortium Blockchain)
负责存储所有域名的公钥凭证和吊销状态，提供具备全局一致性的去中心化 PKI。
* **节点基础设施**：采用 Foundry 体系的 Anvil 本地节点。配置为“即时出块”（Zero Block Time）模式，最大化模拟极高 TPS 的联盟链（如 Hyperledger Fabric 或以太坊 PoA 侧链）的响应特性。
* **数据结构设计 (Registry)**：
  * 核心智能合约维护一个扁平的哈希表映射：`TxID -> (PublicKey, Expiration, RevocationStatus)`。
  * 放弃复杂的默克尔树路径遍历，验证网关仅需单次 `eth_call`（只读调用）即可获取完整凭证，保证区块链 RPC 拉取的时间开销为 `O(1)`。

### 模块三：带外反向 DNS 基础设施 (rDNS Index Source)
作为第一重身份校验和链上数据的引路人，提供极致压缩的认证报文。
* **服务端引擎**：容器化的 BIND9 服务端，监听本地环回地址的 UDP 53 端口。
* **索引结构规范**：
  * 在 `.in-addr.arpa` 逆向解析域中配置自定义 TXT 记录。
  * 记录格式严格控制：协议版本号 + 32字节 `TxID`（哈希指针） + 64字节第一重签名 `Sig1`。总载荷确保低于 150 字节，在实验对比中对标 DNSSEC 动辄数千字节的 RRSIG 记录，形成压倒性优势。

### 模块四：实验编排与网络保真模块 (Experimental Testbed)
为了向顶刊审稿人证明实验环境的严谨性，必须引入网络仿真和极限压测机制。
* **流量发生器**：采用 Rust 编写的自定义 TCP 压测客户端（或 `hping3`），能够精准控制并发连接数（1 到 10,000+），用于绘制系统吞吐量拐点图。
* **网络损伤模拟 (Network Emulation)**：
  * 引入 Linux `tc` (Traffic Control) 结合 `netem` 模块。
  * **策略**：在单机环回网卡上人为注入特定分布的网络延迟（如为 DNS 端口注入 10ms 延迟，为 RPC 端口注入 50ms 延迟）和丢包率，将理想化的单机环境转化为高度拟真的广域网（WAN）跨域交互场景。

## 三、 核心协议执行流水线 (Critical Path)

当一个客户端尝试向目标服务器发起连接时，系统的数据流转严格遵循以下生命周期：



1. **拦截与挂起**：内核 `iptables` 捕获目标出向的 TCP SYN 报文，挂入 Netfilter 队列，网关获取目标 IP。
2. **带外寻址**：网关向 BIND9 发起 `IP.in-addr.arpa` 的 TXT 查询，获取 `TxID` 与 `Sig1`。
3. **第一重防御 ($V_1$)**：网关计算线程验证 `Sig1`。失败则立即下发 `NF_DROP` 阻断连接（防御粗粒度的 IP 伪造）。
4. **状态检索与缓存介入**：
   * 网关以 `TxID` 访问本地 `moka` 缓存。
   * **Cache Miss**：向 Anvil 节点发起只读 RPC 拉取链上 $Cert_{IP}$，并回写缓存。
   * **Cache Hit**：直接进入下一步（消除 RPC 网络开销）。
5. **第二重防御 ($V_2$)**：提取 $Cert_{IP}$ 中的公钥，验证目标服务器的合法性及凭证时效。
6. **裁决释放**：双重验签均通过，网关向内核下发 `NF_ACCEPT`，TCP 握手继续，连接建立。

## 四、 顶刊实验数据保障策略

为了确保你在 Exp-1 到 Exp-5 中拿到预期数据，架构设计已内置了以下保障：
* **对标 Exp-1 (时延拆解)**：系统在流水线的每个阶段（DNS、RPC、$V_1$、$V_2$）均埋入纳秒级探针 (`std::time::Instant`)。由于 Rust 没有垃圾回收期的 Stop-The-World 现象，采集到的拆解时延将能够完美拟合总端到端时延。
* **对标 Exp-2 (报文大小)**：BIND9 的精简 TXT 记录天然规避了 UDP 512 字节截断阈值，这在 Wireshark 抓包数据中将呈现极其直观的断层式对比。
* **对标 Exp-5 (高并发吞吐)**：利用 `tokio` 的轻量级协程（单核可轻松支撑数万并发）配合独立计算线程池，系统的 TPS 瓶颈将被推到极限，仅受限于本机的 CPU 物理核数，而非语言层面的锁机制。

## 五、 实施路径与执行建议

建议按照以下三个阶段进行工程落地：
1. **基建与连通阶段**：部署 BIND9 与 Anvil 节点，写入静态测试数据。编写简单的 Python/curl 脚本验证 DNS 解析和 RPC 调用的连通性。
2. **核心网关开发阶段**：集中精力用 Rust 开发 `Auth Gateway`。先实现 `nfq` 拦截与无条件放行，再逐步接入异步 DNS 查询、缓存模块和密码学验签模块。
3. **仿真与压测阶段**：引入 `tc netem` 设置网络延迟，使用压测工具打满流量。开启网关的性能分析工具（如 `perf` 或 `flamegraph` 火焰图），定位并优化最后的性能热点，导出 CSV 实验数据进行绘图。
