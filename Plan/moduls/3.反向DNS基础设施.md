作为整个双重认证体系的"引路人"与"第一道线索提供方"，**模块三：带外反向 DNS 基础设施（Out-of-Band rDNS Infrastructure）**的设计直接关乎论文的核心卖点——解决传统 DNSSEC 和 DANE 协议中由于证书和签名链过长导致的 UDP 报文截断（UDP Truncation）与 TCP 降级（TCP Fallback）问题。

在 IEEE ToN/TNSM 的审稿标准中，单凭理论推导是不够的。这个模块必须在物理层面上严格构造出**报文大小的断层式优势**（支撑 Exp-2），并提供稳定可控的解析时延基准（支撑 Exp-1）。

以下是该模块的深度架构设计方案：

### 一、 核心创新点：极简 TXT 记录规范

本系统放弃了在 DNS 报文中传输沉重的 X.509 证书或完整的信任链，转而将反向 DNS（rDNS）退化为一个**纯粹的带外索引（Out-of-Band Index）**，仅承载两个关键字段：链上索引指针和节点自签名。

* **数据结构设计**：
    在 `.in-addr.arpa` (IPv4) 或 `.ip6.arpa` (IPv6) 域中，我们定义一种名为 `rOBD` (Reverse Out-of-Band DANE/DNS) 的自定义 TXT 记录。

* **字段定义与字节开销计算**：
    1. **协议标识 (Version)**：`v=rOBD;`（约 7 字节）。
    2. **链上哈希指针 (TxID)**：指向模块二智能合约中 `ChainRecord` 的 32 字节哈希指针。采用 Hex 编码格式：`txid=0x...;`（约 69 字节）。
    3. **节点自签名 (Sig_Sub)**：IP 地址所有者使用其私钥 $SK_{Sub}$ 对 `(IP + TxID)` 进行的**静态预计算签名**。`sig_sub=MEYCIQ...;`（约 90 字节）。

* **`Sig_Sub` 的语义边界（论文中必须明确说明）**：
    * `Sig_Sub` 是节点预计算后存储于 Zone File 中的静态签名，所有查询者共享同一份值。其提供的安全保证是：**证明 IP 地址与 TxID 的绑定关系由持有 $SK_{Sub}$ 的实体声明**，本质上是一种半静态的所有权证明，而非会话级防重放。
    * 该签名被任意一个 DNS 响应嗅探者获取后，在当前 DNS Zone File 有效期（TTL 窗口）内对任意客户端可重放。防重放保护来自两个机制：① DNS TTL 到期后解析缓存失效，Zone File 中的签名随之更新；② 链上凭证吊销机制（秒级生效）。
    * 专利原设计中 $V_2$ 针对会话动态挑战（TCP ISN 或 Nonce）的签名，与此处的静态预计算签名存在工程差异。论文 Discussion 节需明确说明此设计简化的安全假设边界（如适用场景为凭证轮转周期内的稳定网络）。
    * **签名内容选择**：`Sig_Sub = Sign(SK_Sub, IP || TxID)`，去掉 Timestamp 字段，因为 Timestamp 在 Zone File 静态签名场景下不提供额外的会话绑定安全性，反而增加 Zone File 更新频率。

* **架构师论断**：
    单条 TXT 记录的有效载荷（Payload）被严格钉死在约 **170 字节**左右。加上 DNS 报文头和 Question 区域，总 UDP 载荷稳定在 220 字节上下。这距离经典的 512 字节 UDP 截断阈值有极其宽裕的安全余量，从根本上免疫了 TCP 降级带来的 RTT 延迟惩罚。

### 二、 BIND9 服务端架构与配置拓扑

为了在实验中高保真还原真实互联网的 DNS 树状解析体系，我们采用业界绝对标准 **BIND9** 作为服务端引擎，并将其容器化或作为独立守护进程运行在实验台（Testbed）上。

* **拓扑角色**：
    在单机虚拟化环境中，该 BIND9 实例逻辑上扮演目标 IP 所属的**权威反向解析服务器（Authoritative rDNS Server）**。

* **监听地址配置（关键避坑）**：
    BIND9 必须监听 **`127.0.0.2:53`**，**不可使用 `127.0.0.53`**。`127.0.0.53` 是现代 Linux 发行版（Ubuntu 18.04+、Debian Buster+）中 `systemd-resolved` 的默认 stub resolver 监听地址，使用该地址会产生端口占用冲突，导致 BIND9 无法启动或被 `systemd-resolved` 接管。实验初始化脚本中需加入以下操作之一：
    * `sudo systemctl stop systemd-resolved && sudo systemctl disable systemd-resolved`（彻底停止），或
    * 将 `/etc/systemd/resolved.conf` 中的 `DNSStubListener=no` 关闭 stub 监听，再修改网关的 DNS 上游指向 `127.0.0.2:53`。

* **核心配置策略 (Zone Provisioning)**：
    系统无需修改 BIND9 的底层 C 代码，完全通过标准的 Zone File 进行声明式配置。
    以拦截目标 IP `192.168.1.100` 为例，需要在 `/etc/bind/db.192.168.1` 区域文件中静态注入如下资源记录（RR）：
    ```text
    $TTL 86400
    @   IN  SOA ns1.example.net. admin.example.net. ( ... )
        IN  NS  ns1.example.net.
    ; 传统的 PTR 记录保留，兼容现有网络生态
    100 IN  PTR server.example.com.
    ; 注入本文核心的 rOBD TXT 记录（仅含 TxID + Sig_Sub，无 Timestamp）
    100 IN  TXT "v=rOBD; txid=0x7a8b...3f1c; sig_sub=MEYCIQ..."
    ```

### 三、 支撑 Exp-2 的控制组设计 (Baselines)

一篇顶级的系统架构论文，其对比实验（Baselines）必须足够硬核。在这个模块中，你不仅要配置我们自己的方案，还必须**人为构造出 DNSSEC 和 DANE 的臃肿响应**作为反面教材。

1.  **控制组 A：裸 DNSSEC 响应链**
    * **构造方法**：使用 `dnssec-keygen` 和 `dnssec-signzone` 工具对 BIND9 的反向区域进行完整的 NSEC/NSEC3 和 RRSIG 签名，同时在 Zone 中配置 DNSKEY 记录。
    * **预期效果**：当网关请求带有 `DO` (DNSSEC OK) 标志位时，BIND9 返回的响应将附带 `RRSIG` 和 `DNSKEY` 记录。对于包含 RRSIG 和 DNSKEY（ZSK）的典型签名 TXT 响应，实际大小通常在 **500~1200 字节**范围（若同时包含完整 KSK/ZSK 密钥集，可能达到 1500 字节，但这属于特定配置，并非典型情况）。**应以实验实测数据为准，不应在论文中预设超过 1500 字节的期望值**，否则若实测结果与预设差距过大，会导致审稿人质疑实验构造的合理性。建议实际搭建后先进行预实验标定（`dig @127.0.0.2 -p 53 +dnssec 100.1.168.192.in-addr.arpa TXT | wc -c`），以实测数据为准。
2.  **控制组 B：传统 DANE TLSA 记录**
    * **构造方法**：在正向或反向区域中植入 `_443._tcp.server.example.com. IN TLSA ...` 记录，内嵌完整的 X.509 证书哈希或公钥。
    * **预期效果**：报文大小介于本方案与 DNSSEC 之间（约 300~600 字节），在边缘网络环境中仍有一定的截断风险。

### 四、 实验避坑指南：缓存与 TTL 陷阱

在通过脚本自动化跑 Exp-1（时延测量）和 Exp-5（并发吞吐量）时，DNS 模块最容易成为导致数据失真的"内鬼"。

* **Exp-1（时延基准）的缓存策略**：
    * 必须**关闭** `hickory-resolver` 的本地缓存，强制每次 DNS 查询均走真实的 UDP I/O，确保测量到的 rDNS 时延真实反映 DNS 网络开销（加上 `tc netem` 注入的延迟）。

* **Exp-5（吞吐量）的双组对照设计**：
    * **禁用缓存组（最坏情况）**：强制每一个请求向 BIND9 发起真实的 UDP I/O。此时 BIND9 单机 UDP 吞吐量成为瓶颈，测出的 TPS 反映了系统在无 DNS 缓存情况下的绝对下限，同时也是 BIND9 作为系统瓶颈的量化依据。
    * **启用缓存组（真实部署）**：启用合理的 DNS TTL 缓存（对应 Zone File 配置的 TTL，如 3600s）。此时 DNS 开销被摊薄，TPS 瓶颈转移到密码学验签（ECDSA）或 RPC 查询，更真实反映了生产部署场景的性能上限。
    * 论文中需明确说明两组配置的现实含义，避免审稿人认为禁用缓存组是故意制造的极端场景以美化本方案的对比优势。

至此，提供超轻量索引的模块三已经设计完毕。配合高性能的模块一（验证网关）和防篡改的模块二（区块链状态机），整个数据面与控制面的架构已完全闭环。
