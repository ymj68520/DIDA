作为整个双重认证体系的“引路人”与“第一道防线”，**模块三：带外反向 DNS 基础设施（Out-of-Band rDNS Infrastructure）**的设计直接关乎论文的核心卖点——解决传统 DNSSEC 和 DANE 协议中由于证书和签名链过长导致的 UDP 报文截断（UDP Truncation）与 TCP 降级（TCP Fallback）问题。

在 IEEE ToN/TNSM 的审稿标准中，单凭理论推导是不够的。这个模块必须在物理层面上严格构造出**报文大小的断层式优势**（支撑 Exp-2），并提供稳定可控的解析时延基准（支撑 Exp-1）。

以下是该模块的深度架构设计方案：

### 一、 核心创新点：极简 TXT 记录规范

本系统放弃了在 DNS 报文中传输沉重的 X.509 证书或完整的信任链，转而将反向 DNS（rDNS）退化为一个**纯粹的带外索引（Out-of-Band Index）**。



* **数据结构设计**：
    在 `.in-addr.arpa` (IPv4) 或 `.ip6.arpa` (IPv6) 域中，我们定义一种名为 `rOBD` (Reverse Out-of-Band DANE/DNS) 的自定义 TXT 记录。
* **字段定义与字节开销计算**：
    1.  **协议标识 (Version)**：`v=rOBD;` （约 7 字节）。
    2.  **链上哈希指针 (TxID)**：指向模块二智能合约中 $Cert_{IP}$ 的 32 字节哈希。采用 Hex 编码格式：`txid=0x...;` （约 69 字节）。
    3.  **第一重签名 (Sig1 - 防重放增强)**：IP 地址所有者使用其私钥对 `(IP + TxID + Timestamp)` 进行的签名。Timestamp 确保了即便 DNS 响应被截获并重放，网关也能根据过期时间（TTL 感知）进行初步阻断。
* **架构师论断**：
    单条 TXT 记录的有效载荷（Payload）被严格钉死在 **166 字节**左右。加上 DNS 报文头和 Question 区域，总 UDP 载荷稳定在 200 字节上下。这距离经典的 512 字节 UDP 截断阈值有极其宽裕的安全余量，从根本上免疫了 TCP 降级带来的 RTT 延迟惩罚。

### 二、 BIND9 服务端架构与配置拓扑

为了在实验中高保真还原真实互联网的 DNS 树状解析体系，我们采用业界绝对标准 **BIND9** 作为服务端引擎，并将其容器化或作为独立守护进程运行在实验台（Testbed）上。

* **拓扑角色**：
    在单机虚拟化环境中，该 BIND9 实例逻辑上扮演目标 IP 所属的**权威反向解析服务器（Authoritative rDNS Server）**。
* **核心配置策略 (Zone Provisioning)**：
    系统无需修改 BIND9 的底层 C 代码，完全通过标准的 Zone File 进行声明式配置。
    以拦截目标 IP `192.168.1.100` 为例，需要在 `/etc/bind/db.192.168.1` 区域文件中静态注入如下资源记录（RR）：
    ```text
    $TTL 86400
    @   IN  SOA ns1.example.net. admin.example.net. ( ... )
        IN  NS  ns1.example.net.
    ; 传统的 PTR 记录保留，兼容现有网络生态
    100 IN  PTR server.example.com.
    ; 注入本文核心的 rOBD TXT 记录
    100 IN  TXT "v=rOBD; txid=0x7a8b...3f1c; sig1=MEYCIQ..."
    ```
* **网络解耦**：
    配置 BIND9 仅监听特定的隔离网卡（如 `127.0.0.53` 端口 53），确保模块一（Rust 网关）发出的异步 DNS 请求不会污染主机的物理网络或被系统默认的 `systemd-resolved` 劫持。

### 三、 支撑 Exp-2 的控制组设计 (Baselines)

一篇顶级的系统架构论文，其对比实验（Baselines）必须足够硬核。在这个模块中，你不仅要配置我们自己的方案，还必须**人为构造出 DNSSEC 和 DANE 的臃肿响应**作为反面教材。

1.  **控制组 A：裸 DNSSEC 响应链**
    * **构造方法**：使用 `dnssec-keygen` 和 `dnssec-signzone` 工具对 BIND9 的反向区域进行完整的 NSEC/NSEC3 和 RRSIG 签名。
    * **预期效果**：当网关请求带有 `DO` (DNSSEC OK) 标志位时，BIND9 返回的响应不仅包含 TXT 记录，还将附带庞大的 `RRSIG` 和 `DNSKEY` 记录，报文体积将瞬间膨胀至 1500 - 3000 字节，触发实验台抓包工具（Wireshark/Scapy）的 TCP 截断重传。
2.  **控制组 B：传统 DANE TLSA 记录**
    * **构造方法**：在正向或反向区域中植入 `_443._tcp.server.example.com. IN TLSA ...` 记录，内嵌完整的 X.509 证书哈希或公钥。
    * **预期效果**：报文大小介于本方案与全量 DNSSEC 之间（约 300 - 600 字节），在边缘网络环境中仍有极高的截断风险。

### 四、 实验避坑指南：缓存与 TTL 陷阱

在通过脚本自动化跑 Exp-1（时延测量）和 Exp-5（并发吞吐量）时，DNS 模块最容易成为导致数据失真的“内鬼”。

* **TTL 刺客**：在压测 TPS 时，如果 BIND9 返回的 TXT 记录 TTL 设置过长，模块一（Rust 网关）的异步解析器会直接从进程内存中返回结果，导致你的系统看起来拥有无穷大的吞吐量，但审稿人一眼就能看出这是**缓存击穿测试**，缺乏广域网真实性。
* **策略对齐**：必须在网关端（`hickory-resolver`）**强行禁用 DNS 本地缓存**，强制每一个 TCP SYN 握手都向 BIND9 发起真实的 UDP I/O 请求。只有这样，Exp-5 测出的并发瓶颈才是网关处理网络 I/O 的真实瓶颈，数据才经得起推敲。

至此，提供超轻量索引的模块三已经设计完毕。配合高性能的模块一（验证网关）和防篡改的模块二（区块链状态机），整个数据面与控制面的架构已完全闭环。
