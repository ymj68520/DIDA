作为整个双重认证系统的"心脏"，**身份验证网关（Authentication Gateway）**的设计直接决定了你论文中 Exp-1（端到端时延）和 Exp-5（并发吞吐量）的数据上限。

在 Rust 的语境下，我们要彻底抛弃传统多进程/多线程那种"一个请求占用一个线程"的笨重模式，转而采用**"事件驱动 + 异步 I/O + 隔离计算栈"**的现代网关架构。

以下是该模块的深度设计与实现剖析：

### 一、 核心并发与线程模型设计

为了在单机上榨干 CPU 性能并处理成千上万的并发 TCP SYN 握手，我们需要将系统严格划分为两套平行的线程池：

1.  **I/O 反应堆 (Tokio Reactor Pool)**：
    * **职责**：处理所有不消耗 CPU 但需要等待时间的网络操作。包括监听 `NetfilterQueue` 传来的数据包、向 BIND9 发起 UDP DNS 查询、向 Anvil 节点发起 HTTP RPC 请求。
    * **配置**：通常等于机器的逻辑 CPU 核心数。处理过程中需引入 **`In-Flight Task Tracker`**，利用 `moka` 记录活跃的 `(SIP, DIP)`，防御 TCP 重传导致的重复触发。
2.  **密码学计算池 (Spawn Blocking Pool)**：
    * **职责**：专职处理 $V_1$ 和 $V_2$ 的 ECDSA 验签（使用 `k256` 库）以及 Keccak256 哈希计算。
    * **必要性**：一次 ECDSA 验签可能需要数十到数百微秒（$\mu s$）。如果在 I/O 线程中直接计算，会引发"Head-of-Line Blocking（队头阻塞）"，导致后续成百上千的网络包排队超时。因此，必须通过 `tokio::task::spawn_blocking` 将这部分运算"丢"到专门的系统线程中执行。
3.  **内核加速层 (XDP/eBPF - 白名单初筛)**：
    * **职责**：对于**已建立信任的白名单 IP 集合**（即曾经成功通过完整双重验签且缓存仍有效的来源），在内核驱动层执行快速 eBPF MAP 查找，直接放行，极大降低内核态到用户态的上下文切换开销。
    * **重要边界说明**：XDP/eBPF 程序受限于内核 eBPF 验证器的严格约束（指令数上限、禁止复杂循环、内存访问受限），**ECDSA 验签等高计算量密码学操作无法在 XDP 层执行**。XDP 层的作用仅限于比对已有的 IP 白名单集合（通过 `BPF_MAP_TYPE_HASH` 存储），$V_1$ 和 $V_2$ 的密码学验证始终在用户态计算线程池中完成。论文中必须明确此边界，否则会被审稿人质疑为夸大性能声明。

### 二、 关键数据结构与内存共享

在 Rust 中处理极高并发，状态的共享（如缓存和连接池）如果不加锁或者锁粒度不对，会成为致命瓶颈。

```rust
// 网关全局共享状态容器（通过 Arc<GatewayState> 在所有并发任务中传递）
pub struct GatewayState {
    // 1. 高并发内存缓存：存储 TxID -> ChainRecord
    //    Moka 内部已实现极致的分段锁，外部无需加 Mutex
    pub cert_cache: moka::future::Cache<[u8; 32], ChainRecord>,

    // 2. 本地预置的顶级信任锚点公钥 PK_Top
    //    网关启动时从本地配置文件加载，不依赖任何网络查询
    //    是执行 V1 验证的不动根基，整个生命周期只读不写
    pub pk_top: k256::ecdsa::VerifyingKey,

    // 3. 异步 DNS 客户端：内部维护了 UDP socket 池
    pub dns_resolver: hickory_resolver::TokioAsyncResolver,

    // 4. 区块链 RPC 客户端：底层复用 HTTP Keep-Alive 连接池
    pub rpc_client: alloy::providers::ProviderHttp,
}

// 从链上拉取并在本地缓存的完整链上记录
#[derive(Clone)]
pub struct ChainRecord {
    pub cert_ip: CertIP,   // IP 凭证主体（V1 的被验证对象，V2 从中提取 PK_Sub）
    pub sig_top: Vec<u8>,  // 顶级机构授权签名 Sig_Top（V1 的验证素材）
}

// IP 凭证主体（Cert_IP）
#[derive(Clone)]
pub struct CertIP {
    pub ip_prefix: String,                     // 被授权的 IP 前缀
    pub public_key: k256::ecdsa::VerifyingKey, // 凭证持有者公钥 PK_Sub（V2 使用）
    pub expiration_timestamp: u64,             // 凭证到期时间戳（V1 校验）
    pub is_revoked: bool,                      // 吊销状态（V1 校验）
}
```

**架构师点评**：绝不在高频路径上使用标准库的 `std::sync::Mutex`。依靠 `moka` 的并发特性和 HTTP 客户端底层的连接复用，我们实现了状态的"无锁化"读取，这是支撑超高 TPS 的基石。

### 三、 报文处理的微观流水线 (Micro-Pipeline)

当内核通过 `nfq` 抛来一个 TCP SYN 报文时，网关的执行状态机如下：

> **核心原则**：$V_1$ 使用本地预置的 $PK_{Top}$（无需网络查询），$V_2$ 使用从 $Cert_{IP}$ 中提取的 $PK_{Sub}$（需先完成区块链查询）。因此流水线中**区块链查询必须先于两步密码学验证**，正确顺序如下：

1.  **零拷贝提取 (Zero-Copy Extraction)**：
    * 读取 IP Header，提取 `Destination IP`。
    * **关键动作**：立刻丢弃报文 Payload，只保留 `Packet ID` 和 `IP`，将其封装为一个极轻量的任务结构体打入 Tokio 通道（Channel）。不要把整个数据包保留在内存中，防止 OOM（内存溢出）。
2.  **带外 DNS 寻址 (Timeout: 50ms)**：
    * 向 BIND9 发起 `IP.in-addr.arpa` TXT 查询，获取 `TxID` 与 `Sig_Sub`。
    * 这里必须包一层 `tokio::time::timeout`。如果 DNS 查询超时或返回 NXDOMAIN，直接丢弃（Drop）报文，防止资源耗尽。
3.  **状态检索与缓存介入**：
    * 以 `TxID` 访问本地 `moka` 缓存。
    * **Cache Hit**：耗时不到 $1\mu s$，直接拿到 `ChainRecord`，跳过 RPC 调用，进入下一步。
    * **Cache Miss**：通过 `alloy` 向 Anvil 节点发起只读 `eth_call`，拉取完整的 `ChainRecord`（`Cert_IP` + `Sig_Top`）。此步同样需要严格的超时控制。拿到数据后，以异步方式写入 `moka` 缓存。
4.  **第一重防御 $V_1$（权威背书校验）**：
    * 进入 `spawn_blocking` 计算线程池。
    * 使用**本地预置的** $PK_{Top}$（无需任何网络查询）执行：
      $$Verify(PK_{Top},\ Sig_{Top},\ keccak256(Cert_{IP}))$$
    * 同时校验：$T_{current} \le Cert_{IP}.expiration$（凭证未过期）且 $IP_{target} \in Cert_{IP}.ip\_prefix$（IP 归属合法）且 $Cert_{IP}.is\_revoked = false$。
    * **任一条件失败则立即下发 `NF_DROP`**（防御链上凭证被伪造或已吊销）。
5.  **第二重防御 $V_2$（节点控制权校验）**：
    * 从已通过 $V_1$ 校验的 $Cert_{IP}$ 中提取 $PK_{Sub}$（此时已确认该公钥由权威机构背书）。
    * 使用 $PK_{Sub}$ 验证 DNS TXT 记录中的 `Sig_Sub`：
      $$Verify(PK_{Sub},\ Sig_{Sub},\ IP \| TxID)$$
    * **验签失败则下发 `NF_DROP`**（防御持有链上凭证但无私钥控制权的伪冒）。
6.  **异步判决回传**：
    * 拿着最初的 `Packet ID`，通过 FFI 调用底层的 `nfq_set_verdict`，下发 `NF_ACCEPT`。此时内核释放报文，TCP 握手继续，连接建立。
    * **观测埋点**：此时应记录从 `nfq` 捕获到 `verdict` 刷回的总时长，重点观测 **"Kernel-to-User Context Switch Overhead"** 对总体时延的贡献。

### 四、 性能压榨与排坑指南（防挂科必备）

在准备 ToN/TNSM 级别的实验数据时，审稿人会用极其挑剔的眼光审视你的实验设计。在实现这个模块时，必须避开以下陷阱：

* **陷阱 1：DNS 解析器的缓存策略**。
    * *对策*：在进行 Exp-1（单次时延基准测试）时，必须**关闭** `hickory-resolver` 自带的本地缓存，否则你测出的 rDNS 时延会变成 0，这在跨域场景下是不现实的。在进行 Exp-5（高吞吐压测）时，应设计**两组对照**：一组禁用缓存（测量最坏情况，此时 BIND9 成为瓶颈），一组启用缓存（贴近真实部署，测量密码学验签的 TPS 天花板），在论文中分别说明各组的现实含义。

* **陷阱 2：文件描述符（FD）耗尽**。
    * *对策*：高并发下，Linux 默认的 1024 个文件描述符上限会被瞬间打爆。在实验启动脚本中，必须加入 `ulimit -n 65535`，否则网关会因为无法开启新的 UDP Socket（用于 DNS）或 TCP Socket（用于 RPC）而大面积报错。

* **陷阱 3：Netfilter 队列溢出**。
    * *对策*：如果压测工具发包太快，而网关处理不过来，内核的 Netfilter 队列（默认长度 1024）会满，导致内核直接丢包。需要在初始化 `nfq` 时，将 `--queue-maxlen` 参数调大至 `10000` 甚至更高，以吸收瞬时的流量洪峰。

* **陷阱 4：Exp-5 使用 SYN Flood 语义的根本性错误**。
    * *原因*：若压测客户端发送 SYN 后不等待 ACK，TCP 三次握手永远无法完成，网关的 $V_2$ 验签（以及后续的 `NF_ACCEPT` 下发）将永远无法到达终态。这种测试实际只覆盖了流水线的前半段（拦截 + DNS + 缓存查询 + $V_1$），是一个不完整的认证流程压测，测出的 TPS 数字对论文而言不具备有效性。
    * *对策*：Exp-5 的压测客户端**必须完成完整的 TCP 三次握手**，模拟正常的并发认证连接建立场景。真实的性能指标应是"认证成功的完整连接 TPS"和"端到端时延分布（P50/P95/P99）"。

这套设计将验证网关变成了一个高度精密、无阻塞的高速通道，足以在 Exp-5 中跑出极为平滑的 TPS 曲线。
