在双重认证系统的整体架构中，**模块二：联盟链状态机与信任锚点（Consortium Blockchain State Machine and Trust Anchor）**扮演着"全局信任根"的角色。如果说模块一（Rust 网关）是冲锋陷阵的轻骑兵，那么模块二就是稳如泰山的大本营。

为了满足 IEEE ToN/TNSM 级别论文对"去中心化、防篡改、极低查询时延"的严苛证明要求，我们不能把区块链设计得过于沉重。这里的架构核心原则是：**写操作严格准入与审计，读操作极致轻量化与无锁并发。**

以下是该模块的深度架构设计方案：

### 一、 核心智能合约设计：完整信任链存储

智能合约的唯一职责是作为 `Cert_IP` 凭证与顶级授权签名的权威键值存储。**任何丢失 `Sig_Top` 字段的设计都会使双重信任链名存实亡**——若链上不存储 `Sig_Top`，任何拥有合约写入权限的节点均可将任意公钥写入 `TxID` 映射，而验证网关对此无从察觉，整个系统退化为无信任根的普通 PKI。

* **开发语言与版本**：Solidity `^0.8.20`（利用最新的自定义错误和 Gas 优化特性）。
* **数据结构（Data Schema）**：
    * **核心映射（Mapping）**：使用 `mapping(bytes32 => ChainRecord) public registry;`。这里的 `bytes32` 就是网关从 DNS TXT 记录中拿到的 `TxID`（哈希指针）。
    * **顶层结构（Struct `ChainRecord`）**：
        * `certIP`（`CertIP` 类型）：IP 凭证主体，是 $V_1$ 中被验证的完整数据对象。
        * `sigTop`（`bytes` 类型）：**顶级权威机构**（如 IANA/RIR 对应的联盟链权威节点）对 `keccak256(abi.encode(certIP))` 的 secp256k1 ECDSA 签名，即 $Sig_{Top} = Sign(SK_{Top}, Hash(Cert_{IP}))$。这是 $V_1$ 的核心被验证素材，缺失此字段则 $V_1$ 无从执行。
    * **凭证主体（Struct `CertIP`）**：
        * `ipPrefix`（`string`）：被授权的 IP 前缀（如 `"192.168.1.0/24"`），$V_1$ 需确认目标 IP 属于此前缀。
        * `publicKey`（`bytes`）：凭证持有者的 secp256k1 公钥 $PK_{Sub}$，$V_2$ 使用此字段验证 `Sig_Sub`。
        * `expiration`（`uint64`）：凭证到期时间戳，$V_1$ 需校验 $T_{current} \le expiration$。
        * `isRevoked`（`bool`）：吊销状态位（一键阻断被黑客盗用的密钥）。

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract IPCertRegistry {
    struct CertIP {
        string  ipPrefix;       // IP 前缀，如 "192.168.1.0/24"
        bytes   publicKey;      // 凭证持有者公钥 PK_Sub (secp256k1, 65字节非压缩格式)
        uint64  expiration;     // 凭证过期 Unix 时间戳
        bool    isRevoked;      // 吊销状态
    }

    struct ChainRecord {
        CertIP  certIP;         // IP 凭证主体（被签名对象）
        bytes   sigTop;         // 顶级机构对 keccak256(certIP) 的授权签名 Sig_Top
    }

    // TxID (bytes32) -> 完整链上记录
    mapping(bytes32 => ChainRecord) public registry;

    // 吊销事件，供网关 WebSocket 订阅
    event Revoked(bytes32 indexed txId);
    // 凭证注册事件
    event CertRegistered(bytes32 indexed txId, string ipPrefix);

    // 管理面：注册凭证（需由顶级权威节点签名后，由授权账户调用）
    function registerCert(
        bytes32 txId,
        CertIP calldata certIP,
        bytes  calldata sigTop
    ) external onlyAuthorized {
        registry[txId] = ChainRecord({ certIP: certIP, sigTop: sigTop });
        emit CertRegistered(txId, certIP.ipPrefix);
    }

    // 管理面：触发吊销
    function revokeCert(bytes32 txId) external onlyAuthorized {
        registry[txId].certIP.isRevoked = true;
        emit Revoked(txId);
    }

    // 修饰器占位（论文实验阶段可简化为合约部署者地址校验）
    modifier onlyAuthorized() { _; }
}
```

* **时间复杂度说明**：
    MPT 状态树查找的严格复杂度为 $O(k)$，其中 $k$ 是键的长度（字节数）。由于 `TxID` 为固定长度的 32 字节哈希（64 个十六进制字符，路径深度固定），$k$ 为常数，因此查询路径深度与链上状态总量（无论 1 万条还是 1 亿条）无关，整体查询复杂度与数据规模完全解耦，表现为常数级别。这是支撑论文 Exp-3（常数级查找时间）的核心论点，表述方式需严格如此，以防熟悉 MPT 原理的审稿人以"O(k) ≠ O(1)"为由质疑论文严谨性。

### 二、 底层节点基础设施：Foundry Anvil

为了在单机上模拟真实联盟链的高并发与低时延（支撑 Exp-5），必须摒弃传统的 Ganache 或厚重的 Geth 节点，转而采用纯 Rust 编写的 **Foundry Anvil** 节点。

* **运行模式：即时出块（Zero Block Time）**
    * 在真实广域网联盟链中，共识机制（如 PBFT 或 PoA）会引入秒级的写延迟，但这**不影响读延迟**。
    * 网关的查询操作本质上是无状态变更的只读调用（`eth_call`），不需要打包进区块。因此，我们在本地启动 Anvil 时配置 `--block-time 0`，使其专注于全速响应 HTTP JSON-RPC 查询，消除人为的区块挖掘等待时间。
    * **局限性声明（论文 Discussion 节必须提及）**：Anvil 本地节点的 `eth_call` 时延仅反映了本地内存查找开销，不包含生产级联盟链（如 Hyperledger Besu PoA）共识协议引入的额外只读查询延迟。建议在 Discussion 节提供参数化的延迟补偿模型（如基于 Besu 的基准测试数据叠加网络损伤模拟），使实验结论具备外推性。
* **并发连接调优**：
    * Anvil 默认的并发连接池非常深。为了配合网关的极限压测，无需设置最大连接数限制，只需确保操作系统的 socket 端口不被耗尽（配合上文提到的 `ulimit` 调整）。

### 三、 读写分离的交互接口 (RPC Interface)

系统必须明确划分"管理面"和"数据面"的交互路径，这也是网络安全顶刊非常看重的架构严谨性。

1. **管理面（慢速，写状态，需消耗 Gas）**：
    * **动作**：权威机构管理员注册新的 IP 凭证（含 `Cert_IP` + `Sig_Top`）、更新或触发吊销（Revoke）。
    * **路径**：通过 `eth_sendRawTransaction` 发起状态变更交易，需要私钥签名，等待节点共识并打包入块。在论文实验中，这部分发生在系统初始化阶段，不在性能测量的关键路径（Critical Path）上。
2. **数据面（极速，读状态，零 Gas 开销）**：
    * **动作**：Rust 身份验证网关在缓存未命中时，拉取指定 `TxID` 的完整 `ChainRecord`。
    * **路径**：通过 `eth_call` 触发，单次调用返回 `(CertIP, sigTop)` 完整结构。
    * **撤销实时性 (Real-time Revocation)**：
        * **策略一：短 TTL + 背景续租**：缓存有效期设短（如 30s），但采用 `moka` 的 `refresh_after_write` 异步发起更新。
        * **策略二：事件监听 (Event Watcher)**：网关通过 WebSocket 订阅 `Revoked(bytes32 txId)` 事件。一旦触发吊销，立即主动驱逐 Cache，将撤销延迟从分钟级降至秒级。

### 四、 针对论文实验的对齐与排坑

在实现此模块时，需重点关注以下细节，以确保实验数据的完美呈现：

* **应对 Exp-3（存储规模与时延关系）**：
    * 你需要编写一个自动化的 Solidity 部署与灌水脚本（使用 Foundry 的 `Forge` 工具链）。
    * 分别向合约中注入 100 条、1,000 条、10,000 条模拟的 `ChainRecord`（每条包含完整的 `CertIP` + `sigTop`）。随后使用网关发起随机查询，证明 RPC 拉取时延的分布是一条完美的水平线（不受数据量增加的影响）。
* **网络损伤隔离（关键防挂科点）**：
    * 在 Exp-1 中，当你使用 `tc netem` 模拟广域网延迟时，**切记要对不同的端口实施不同的延迟策略**。
    * 例如：DNS 端口（53）设置 20ms 延迟，区块链 RPC 端口（8545）设置 50ms 延迟。如果对整个 `lo`（环回网卡）无差别加延迟，会导致 `eth_call` 的握手过程被重复惩罚，你的时延分解图（Pie Chart）就会严重失真。
* **网关侧对应的 Rust 数据结构**：

```rust
// 网关全局共享状态容器（通过 Arc<GatewayState> 在所有并发任务中传递）
pub struct GatewayState {
    // 高并发内存缓存：存储 TxID -> ChainRecord
    pub cert_cache: moka::future::Cache<[u8; 32], ChainRecord>,
    // 本地预置的顶级信任锚点公钥（网关启动时从配置文件加载，无需网络查询）
    pub pk_top: k256::ecdsa::VerifyingKey,
    // 异步 DNS 客户端
    pub dns_resolver: hickory_resolver::TokioAsyncResolver,
    // 区块链 RPC 客户端（底层复用 HTTP Keep-Alive 连接池）
    pub rpc_client: alloy::providers::ProviderHttp,
}

// 从链上拉取并在本地缓存的完整记录
#[derive(Clone)]
pub struct ChainRecord {
    pub cert_ip: CertIP,   // IP 凭证主体（V1 的被验证对象）
    pub sig_top: Vec<u8>,  // 顶级机构授权签名（V1 的验证素材）
}

// IP 凭证主体（Cert_IP）
#[derive(Clone)]
pub struct CertIP {
    pub ip_prefix: String,                    // 被授权的 IP 前缀
    pub public_key: k256::ecdsa::VerifyingKey, // 凭证持有者公钥 PK_Sub（V2 使用）
    pub expiration_timestamp: u64,            // 凭证到期时间戳
    pub is_revoked: bool,                     // 吊销状态
}
```

模块二作为去中心化的信任底座，其设计必须同时保证两点：**完整存储权威背书签名**（不可丢失 `Sig_Top`）以及**极致的读取性能**（单次 `eth_call`，常数级时延）。两者缺一不可。
