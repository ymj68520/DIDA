在双重认证系统的整体架构中，**模块二：联盟链状态机与信任锚点（Consortium Blockchain State Machine and Trust Anchor）**扮演着“全局信任根”的角色。如果说模块一（Rust 网关）是冲锋陷阵的轻骑兵，那么模块二就是稳如泰山的大本营。

为了满足 IEEE ToN/TNSM 级别论文对“去中心化、防篡改、极低查询时延”的严苛证明要求，我们不能把区块链设计得过于沉重。这里的架构核心原则是：**写操作严格准入与审计，读操作极致轻量化与无锁并发。**

以下是该模块的深度架构设计方案：

### 一、 核心智能合约设计：极简状态机

智能合约（Smart Contract）不应包含复杂的业务逻辑或多重循环，它的唯一职责是作为 `$Cert_{IP}$` 的键值对数据库。



* **开发语言与版本**：Solidity `^0.8.20`（利用最新的自定义错误和 Gas 优化特性）。
* **数据结构（Data Schema）**：
    * **核心映射（Mapping）**：使用 `mapping(bytes32 => CertIP) public registry;`。这里的 `bytes32` 就是网关从 DNS TXT 记录中拿到的 `TxID`（哈希指针）。
    * **实体结构（Struct `CertIP`）**：
        * `owner` (address)：该 IP 段所属 AS（自治系统）的链上身份凭证。
        * `publicKey` (bytes)：用于第二重验签（$V_2$）的 secp256k1 原始公钥。
        * `expiration` (uint64)：时间戳，网关拉取后在本地比对，防止凭证过期。
        * `isRevoked` (bool)：吊销状态位（一键阻断被黑客盗用的公钥）。
* **时间复杂度保障**：
    * 审稿人非常关注链上检索的开销。通过扁平的 `mapping` 结构，无论链上沉淀了 1 万条还是 1 亿条 IP 凭证，底层 EVM（以太坊虚拟机）通过 Merkle Patricia Trie 的状态树寻址，时间复杂度始终保持在常数级别 $O(1)$。这是支撑论文 Exp-3（常数级查找时间）的核心论点。

### 二、 底层节点基础设施：Foundry Anvil

为了在单机上模拟真实联盟链的高并发与低时延（支撑 Exp-5），必须摒弃传统的 Ganache 或厚重的 Geth 节点，转而采用纯 Rust 编写的 **Foundry Anvil** 节点。



* **运行模式：即时出块（Zero Block Time）**
    * 在真实广域网联盟链中，共识机制（如 PBFT 或 PoA）会引入秒级的写延迟，但这**不影响读延迟**。
    * 网关的查询操作本质上是无状态变更的只读调用（`eth_call`），不需要打包进区块。因此，我们在本地启动 Anvil 时配置 `--block-time 0`，使其专注于全速响应 HTTP JSON-RPC 查询，消除人为的区块挖掘等待时间。
* **并发连接调优**：
    * Anvil 默认的并发连接池非常深。为了配合网关的极限压测，无需设置最大连接数限制，只需确保操作系统的 socket 端口不被耗尽（配合上文提到的 `ulimit` 调整）。

### 三、 读写分离的交互接口 (RPC Interface)

系统必须明确划分“管理面”和“数据面”的交互路径，这也是网络安全顶刊非常看重的架构严谨性。

1.  **管理面（慢速，写状态，需消耗 Gas）**：
    * **动作**：AS 管理员注册新的 IP 凭证、更新公钥或触发吊销（Revoke）。
    * **路径**：通过 `eth_sendRawTransaction` 发起状态变更交易，需要私钥签名，等待节点共识并打包入块。在论文实验中，这部分发生在系统初始化阶段，不在性能测量的关键路径（Critical Path）上。
2.  **数据面（极速，读状态，零 Gas 开销）**：
    * **动作**：Rust 身份验证网关在缓存未命中时，拉取指定 `TxID` 的状态。
    * **路径**：通过 `eth_call` 触发。
    * **撤销实时性 (Real-time Revocation)**：
        * **策略一：短 TTL + 背景续租**：缓存有效期设短（如 30s），但采用 `moka` 的 `refresh_after_write` 异步发起更新。
        * **策略二：事件监听 (Event Watcher)**：网关通过 WebSocket 订阅 `Revoked(bytes32 txid)` 事件。一旦触发吊销，立即主动驱逐 Cache，将撤销延迟从分钟级降至秒级。

### 四、 针对论文实验的对齐与排坑

在实现此模块时，需重点关注以下细节，以确保实验数据的完美呈现：

* **应对 Exp-3（存储规模与时延关系）**：
    * 你需要编写一个自动化的 Solidity 部署与灌水脚本（使用 Foundry 的 `Forge` 工具链）。
    * 分别向合约中注入 100 条、1,000 条、10,000 条模拟的 $Cert_{IP}$ 记录。随后使用网关发起随机查询，证明 RPC 拉取时延的分布是一条完美的水平线（不受数据量增加的影响）。
* **网络损伤隔离（关键防挂科点）**：
    * 在 Exp-1 中，当你使用 `tc netem` 模拟广域网延迟时，**切记要对不同的端口实施不同的延迟策略**。
    * 例如：DNS 端口（53）设置 20ms 延迟，区块链 RPC 端口（8545）设置 50ms 延迟。如果对整个 `lo`（环回网卡）无差别加延迟，会导致 `eth_call` 的握手过程被重复惩罚，你的时延分解图（Pie Chart）就会严重失真。

模块二作为去中心化的信任底座，其设计极其克制，一切为了极致的读取性能让路。
