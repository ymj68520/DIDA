这是整个系统闭环的最后一战。作为架构师，我必须提醒你：在 IEEE ToN 或 TNSM 这样的顶刊评审中，**"实验环境的保真度"往往比"协议本身的设计"更容易受到审稿人的攻击。**

如果你只是在单机 `127.0.0.1` 上跑通了代码，测出零点几毫秒的时延，审稿人会毫不留情地以"缺乏广域网（WAN）真实性"为由拒稿。**模块四：实验编排与网络保真模块（Experimental Testbed & Network Emulation）**的存在，就是为了在单机（或局域网）环境下，完美"伪造"出复杂的真实互联网拓扑、延迟和网络拥塞，并提供手术刀级别的数据采集能力。

以下是该模块的深度架构设计方案：

### 一、 核心底座：高保真网络损伤注入 (Network Emulation)

这个子模块的职责是打破单机环回网卡（Loopback）过于理想的物理限制，为人为制造跨域认证的真实环境。

* **底层引擎**：Linux 内核自带的 `tc` (Traffic Control) 与 `netem` (Network Emulator) 排队规则（Qdisc）。
* **精细化路由与端口隔离策略**：
    绝对不能对整张网卡无差别注入延迟，必须基于目标端口（iptables + fwmark）进行精确打击。
    * **针对 DNS 通道 (UDP 53)**：模拟从本地递归节点到根/顶级域再到权威反向节点的 RTT。使用 `tc` 注入 **$20 \sim 50$ ms 的正态分布延迟（Delay with Jitter）**。
    * **针对区块链 RPC 通道 (TCP 8545)**：模拟跨洲际调用或拥堵的区块链网络。注入 **$50 \sim 150$ ms 的延迟，并附加 $0.1\%$ 的随机丢包率（Packet Loss）**。
    * **针对目标 TCP SYN（模拟 Exp-1 的 Baseline）**：保持极低延迟，作为裸奔对照组。

* **`tc netem` 与 `iptables NFQUEUE` 规则共存的优先级配置**：
    两套规则共享同一个 `iptables` 规则链，若配置顺序不当，存在 NFQUEUE 拦截的 SYN 报文在用户态处理期间与 tc 队列的延迟注入时机叠加，导致测量到的 RPC 时延包含意外的多次延迟惩罚。
    **推荐配置顺序（必须严格遵守）**：
    1. **先配置 `tc` 规则**：通过 `iptables -t mangle -A OUTPUT -p udp --dport 53 -j MARK --set-mark 1` 给 DNS 包打 fwmark，通过 `iptables -t mangle -A OUTPUT -p tcp --dport 8545 -j MARK --set-mark 2` 给 RPC 包打 fwmark，再通过 `tc` 的 `filter` 规则将对应 fwmark 的包导向对应的 `netem` qdisc。
    2. **后配置 `NFQUEUE` 规则**：`iptables -A FORWARD -p tcp --syn -j NFQUEUE --queue-num 0`。
    3. **基准验证**：配置完成后，先使用 `ping` 和 `hping3 -S` 各发 100 个包，用 `tcpdump` 抓取时间戳，验证 DNS 延迟和 RPC 延迟的注入是否符合预期，且不存在叠加效应，然后再正式开始实验数据采集。

* **架构师的底气**：
    通过这种精细的隔离，你可以在论文的 Section VI (Evaluation) 中理直气壮地声明："尽管实验部署在单台高配服务器上，但核心网络交互的 RTT 均已通过 Linux TC 严格对齐了跨国 WAN 环境，消除了本地 IPC 通信带来的虚假红利。"

### 二、 极限流量压测引擎 (Extreme Load Generator)

为了在 Exp-5（高并发场景下的认证吞吐量测试）中探寻我们 Rust 网关的真实 TPS（每秒事务处理量）天花板，常规的测试工具是不够的。

* **流量发生器选型与语义正确性（关键）**：
    * **主攻手：定制化 Rust 并发 TCP 客户端**。采用 Rust `tokio` 异步运行时，同时向目标 IP 发起大量**完整 TCP 三次握手**的并发连接。
    * **为什么必须完成完整的三次握手**：网关的 $V_2$ 验签发生在 TCP 会话上下文确立后，`NF_ACCEPT` 的下发也在 $V_2$ 通过之后。若客户端发送 SYN 后不等待 ACK（即 SYN Flood 模式），三次握手永远无法完成，$V_2$ 永远无法到达终态，`NF_ACCEPT` 无从下发。这种测试实际只覆盖了流水线的前半段（拦截 + DNS + $V_1$），测出的数字对于"完整认证吞吐量"而言**不具备有效性**，审稿人一眼可以识破。
    * **正确的并发模型**：客户端与目标建立完整的 TCP 连接（发 SYN → 收 SYN-ACK → 发 ACK），连接建立后可立即关闭（FIN），连续循环以制造持续的连接洪峰。利用 `tokio::spawn` 同时维持 1 到 10,000+ 个并发连接，精准控制并发度，用于绘制系统吞吐量拐点图。

* **资源监控与瓶颈定位**：
    * 压测期间，后台必须挂载 `htop` 和 `perf`（或生成 Flamegraph 火焰图）。
    * 审稿人喜欢看"失败分析"。你需要在这个模块中动态调优，直到定位到网关的崩溃点究竟是 CPU 算力（ECDSA 验签耗尽 CPU 核）、内存（并发 Cache 打爆）、还是文件描述符（Socket 耗尽），并在论文中诚实地画出吞吐量拐点图。

### 三、 手术刀级遥测与数据采集 (Precision Telemetry)

数据不能靠"大概"估算，必须彻底消除"观察者效应（Observer Effect）"——即测量工具本身不能拖累系统的性能。

* **针对 Exp-1（端到端时延基准测试）与 Exp-4（计算开销）**：
    * **内部微观探针**：基于 Rust `tracing` 生态。利用 `tracing-flamegraph` 记录 CPU 周期分布。重点计算从 `nfq` 读取到 $V_1$ 完成、从 RPC 返回到 $V_2$ 完成等各阶段耗时。
    * **内核切换监测**：专门配置一个探针，利用 `ebpf` 挂载点（如 `tracepoint:nf_queue:nfq_dequeue`），采集报文在内核队列停留的确切时长，对比网关用户态耗时，揭示架构的系统级开销。
    * **落盘策略**：测试期间，探针数据写入内存队列，测试结束后一次性 Flush 到 CSV 文件中，绝对禁止在处理报文的高频路径上执行同步的 Disk I/O（打印日志）。

* **针对 Exp-2（DNS 报文大小对比）**：
    * **外部宏观旁路**：使用 `tcpdump` 在后台静默抓取 UDP 53 端口的 `pcap` 包。
    * 后续使用 `tshark` 或 `Scapy` 脚本离线分析 `.pcap` 文件，精准统计出本文方案 TXT 记录（约 220 Bytes）、DNSSEC（实测值，通常 500~1200 Bytes）和 DANE（约 300~600 Bytes）的载荷分布，生成极具视觉冲击力的柱状图。
    * **重要**：DNSSEC 和 DANE 的实际报文大小以预实验实测数据为准，不应在代码或论文中硬编码预期值。

* **针对 Exp-4 (Ablation Study 消融实验设计)**：
    * **中间件架构 (Middleware Layering)**：网关代码采用 `Layer` 模式封装校验逻辑。测试时可一键关闭"缓存层"、"$V_2$ 校验层"或"异步 RPC 层"，横向对比各模块对吞吐量的实际边际贡献。

### 四、 Exp-5 专项配置矩阵

鉴于 Exp-5 涉及多个变量，需明确设计配置矩阵，确保每组实验的含义清晰可比：

| 配置组 | DNS 缓存 | 压测客户端 | 衡量目标 |
|--------|---------|-----------|---------|
| **组A（最坏情况）** | 禁用 | 完整 TCP 握手 | 无缓存时完整认证流程 TPS 下限（BIND9 成为瓶颈） |
| **组B（真实部署）** | 启用（TTL 3600s）| 完整 TCP 握手 | 真实部署场景下的认证 TPS 上限（密码学/RPC 成为瓶颈） |
| **组C（消融：无 $V_2$）** | 启用 | 完整 TCP 握手 | $V_2$ 模块对 TPS 的边际代价 |
| **组D（消融：无缓存层）** | N/A（禁用）| 完整 TCP 握手 | 缓存模块对 TPS 的边际贡献 |

### 五、 CI/CD 式自动化编排流 (Automated Orchestration)

一个严谨的顶刊实验，必须是一键可复现（Reproducible）的。你需要设计一个 `Makefile` 或 `bash` 编排脚本来统御全局：

1.  **`make setup`**：
    * 停止 `systemd-resolved`（或禁用其 stub listener）；
    * 拉起 BIND9（监听 `127.0.0.2:53`）并静默部署 Solidity 合约（灌入 10,000 条测试数据，每条含完整 `ChainRecord`）；
    * 拉起 Anvil 节点；
    * **先**应用 `tc netem` 损伤规则（`fwmark` 分端口延迟注入），**后**配置 `iptables NFQUEUE` 拦截规则；
    * 通过基准验证脚本（ping + hping3）确认延迟注入隔离效果。
2.  **`make run_exp1_to_5`**：按顺序拉起 Rust 网关（分别以不同参数配置），调用完整握手压测引擎打流，并在每个实验结束后将原始数据归档到 `/results/exp_X/` 目录下。
3.  **`make plot`**：调用 Python 的 `matplotlib`/`seaborn` 脚本，读取归档的 CSV 数据，自动渲染出符合 IEEE 标准的高清矢量图（PDF 格式）。
4.  **`make clean`**：销毁测试环境，清除 `iptables` NFQUEUE 规则和 `tc netem` 规则，释放端口，恢复 `systemd-resolved`。
