这是整个系统闭环的最后一战。作为架构师，我必须提醒你：在 IEEE ToN 或 TNSM 这样的顶刊评审中，**“实验环境的保真度”往往比“协议本身的设计”更容易受到审稿人的攻击。**

如果你只是在单机 `127.0.0.1` 上跑通了代码，测出零点几毫秒的时延，审稿人会毫不留情地以“缺乏广域网（WAN）真实性”为由拒稿。**模块四：实验编排与网络保真模块（Experimental Testbed & Network Emulation）**的存在，就是为了在单机（或局域网）环境下，完美“伪造”出复杂的真实互联网拓扑、延迟和网络拥塞，并提供手术刀级别的数据采集能力。

以下是该模块的深度架构设计方案：

### 一、 核心底座：高保真网络损伤注入 (Network Emulation)

这个子模块的职责是打破单机环回网卡（Loopback）过于理想的物理限制，为人为制造跨域认证的真实环境。



* **底层引擎**：Linux 内核自带的 `tc` (Traffic Control) 与 `netem` (Network Emulator) 排队规则（Qdisc）。
* **精细化路由与端口隔离策略**：
    绝对不能对整张网卡无差别注入延迟，必须基于目标端口（iptables + fwmark）进行精确打击。
    * **针对 DNS 通道 (UDP 53)**：模拟从本地递归节点到根/顶级域再到权威反向节点的 RTT。使用 `tc` 注入 **$20 \sim 50$ ms 的正态分布延迟（Delay with Jitter）**。
    * **针对 区块链 RPC 通道 (TCP 8545)**：模拟跨洲际调用或拥堵的区块链网络。注入 **$50 \sim 150$ ms 的延迟，并附加 $0.1\%$ 的随机丢包率（Packet Loss）**。
    * **针对 目标 TCP SYN (模拟 Exp-1 的 Baseline)**：保持极低延迟，作为裸奔对照组。
* **架构师的底气**：
    通过这种精细的隔离，你可以在论文的 Section VI (Evaluation) 中理直气壮地声明：“尽管实验部署在单台高配服务器上，但核心网络交互的 RTT 均已通过 Linux TC 严格对齐了跨国 WAN 环境，消除了本地 IPC 通信带来的虚假红利。”

### 二、 极限流量压测引擎 (Extreme Load Generator)

为了在 Exp-5（高并发场景下的认证吞吐量测试）中探寻我们 Rust 网关的真实 TPS（每秒事务处理量）天花板，常规的测试工具是不够的。

* **流量发生器选型**：
    * **主攻手：定制化 Rust 并发客户端 / `hping3`**。抛弃 Python 的 `requests` 或 `socket` 脚本。我们需要在极短时间内向目标 IP 轰出成千上万个 `TCP SYN` 包。
    * **并发模型**：采用多线程异步非阻塞发包，绕过操作系统的 TCP 握手状态机限制（即发送 SYN 后不等待 ACK，只管发），以此制造极致的“半连接洪峰”（SYN Flood）。
* **资源监控与瓶颈定位**：
    * 压测期间，后台必须挂载 `htop` 和 `perf`（或生成 Flamegraph 火焰图）。
    * 审稿人喜欢看“失败分析”。你需要在这个模块中动态调优，直到定位到网关的崩溃点究竟是 CPU 算力（ECDSA 验签耗尽 CPU 核）、内存（并发 Cache 打爆）、还是文件描述符（Socket 耗尽），并在论文中诚实地画出吞吐量拐点图。

### 三、 手术刀级遥测与数据采集 (Precision Telemetry)

数据不能靠“大概”估算，必须彻底消除“观察者效应（Observer Effect）”——即测量工具本身不能拖累系统的性能。



* **针对 Exp-1（端到端时延基准测试）与 Exp-4（计算开销）**：
    * **内部微观探针**：基于 Rust `tracing` 生态。利用 `tracing-flamegraph` 记录 CPU 周期分布。重点计算从 `nfq` 读取到 $V_1$ 完成、从 RPC 返回到 $V_2$ 完成等各阶段耗时。
    * **内核切换监测**：专门配置一个探针，利用 `ebpf` 挂载点（如 `tracepoint:nf_queue:nfq_dequeue`），采集报文在内核队列停留的确切时长，对比网关用户态耗时，揭示架构的系统级开销。
    * **落盘策略**：测试期间，探针数据写入内存队列，测试结束后一次性 Flush 到 CSV 文件中，绝对禁止在处理报文的高频路径上执行同步的 Disk I/O（打印日志）。
* **针对 Exp-2（DNS 报文大小对比）**：
    * **外部宏观旁路**：使用 `tcpdump` 在后台静默抓取 UDP 53 端口的 `pcap` 包。
    * 后续使用 `tshark` 或 `Scapy` 脚本离线分析 `.pcap` 文件，精准统计出本文方案 TXT 记录（约 150 Bytes）、DNSSEC（> 1500 Bytes）和 DANE（约 500 Bytes）的载荷分布，生成极具视觉冲击力的柱状图。
* **针对 Exp-4 (Ablation Study 消融实验设计)**：
    * **中间件架构 (Middleware Layering)**：网关代码采用 `Layer` 模式封装校验逻辑。测试时可一键关闭“缓存层”、“$V_2$ 校验层”或“异步 RPC 层”，横向对比各模块对吞吐量的实际边际贡献。

### 四、 CI/CD 式自动化编排流 (Automated Orchestration)

一个严谨的顶刊实验，必须是一键可复现（Reproducible）的。你需要设计一个 `Makefile` 或 `bash` 编排脚本来统御全局：

1.  **`make setup`**：拉起 Anvil 节点并静默部署 Solidity 合约（灌入 10,000 条测试数据）；拉起 BIND9 容器；应用 `tc netem` 损伤规则。
2.  **`make run_exp1_to_5`**：按顺序拉起 Rust 网关（分别以不同的参数配置，如开关 DNSSEC 对照组、开关 Cache），调用压测引擎打流，并在每个实验结束后将原始数据归档到 `/results/exp_X/` 目录下。
3.  **`make plot`**：调用 Python 的 `matplotlib`/`seaborn` 脚本，读取归档的 CSV 数据，自动渲染出符合 IEEE 标准的高清矢量图（PDF 格式）。
4.  **`make clean`**：销毁测试环境，清除 iptables 和 tc 规则，释放端口。
